# Reference Observation v0.1

## RFM workflow summary

Core route:
1. Find at least three strong non-SaaS references.
2. Pick one lead reference, with one or two support references.
3. Capture and observe a real screenshot of the lead reference.
4. Decompose composition, whitespace, material, typography, density, and non-uniformity.
5. Record borrow / do-not-borrow and avoid / aim lists.
6. Decide materials before layout.
7. Build one artifact, render screenshots, and self-evaluate.

Success factors named by the validation summary:
- Committing to one lead reference instead of averaging several.
- Writing the avoid / aim list before building.
- Deciding fonts and hero material before layout.
- Shipping one concrete `index.html`, not stopping at candidates.
- Looking at rendered screenshots and evaluating honestly.

Common weaknesses named by the validation summary:
- Same-kind elements can still become templated in treatment.
- Past runs observed references from known design language more than fresh live capture.
- The page can start from a named concept instead of starting from visual language.
- Lower sections can weaken into ordinary information layout if not guarded.

## Reference candidates

### 1. Space Jam 1996

URL: https://www.spacejam.com/1996/

Why it is useful:
- It is not a SaaS LP and has no modern hero stack.
- The first view is a navigational constellation, not a sales argument.
- The logo and destinations form an irregular spatial system with lots of empty star-field around it.
- Labels are attached to image objects rather than placed in equal cards.

Borrow:
- A dense central object surrounded by irregular destination nodes.
- A screen that behaves like a map / kiosk menu.
- Image-first navigation, small labels, black field, scattered focus.
- Uneven object scale and spacing.

Do not borrow:
- The Space Jam logo, characters, planets, sports theme, or exact radial placement.
- Pure novelty nostalgia, animated GIF feel, or a simple star wallpaper joke.
- Red legal footer, trademark material, or copyrighted imagery.

### 2. Cameron's World

URL: https://www.cameronsworld.net/

Why it is useful:
- It treats the page as a dense excavated collage rather than a clean funnel.
- It uses many small artifacts, unpredictable density, and unpolished image/text combinations.
- It resists the equal-card information grid.

Borrow:
- Collage density, wildly different small materials, abrupt labels, and artifact accumulation.
- The feeling that every item has a slightly different origin.

Do not borrow:
- The exact GeoCities graphics, copied text, novelty maximalism, or chaotic unreadability.
- Any archived third-party images as page assets.

### 3. Windows 93

URL: https://www.windows93.net/

Why it is useful:
- It is an experimental interface object rather than a conventional webpage.
- It uses a dense environment with many affordances and a strange software logic.

Borrow:
- The idea that a web page can be a simulated device / environment.
- Compact controls, status surfaces, and odd affordances.

Do not borrow:
- Windows desktop mimicry, app icons, start menu jokes, fake OS chrome, or parody software tropes.

### 4. Telehack

URL: https://telehack.com/

Why it is useful:
- It is a command interface / simulated historical network, not a landing page.
- It uses sparse textual density and a strong machine register.

Borrow:
- Command-line fragments, machine prompts, serial numbers, and terminal-like status language.

Do not borrow:
- Full terminal screen layout, green-on-black monoculture, or text-only austerity.

## Lead reference

Lead: Space Jam 1996.

Support references: Cameron's World for material density, Telehack for machine-language texture.

Reason for lead choice:
- It is closest to the requested early multimedia / strange interface register.
- The first view gives a clear composition to mimic without copying: central identity object, irregular surrounding nodes, star-field emptiness, and image labels.
- Its weakness is also useful: if copied literally it becomes cheap retro, so the design must translate the structure into a different information object.

## Lead screenshot

Captured screenshot: `notes/lead_reference_spacejam_1996.png`

Observed first-view composition:
- A black star-field fills the entire viewport.
- The main logo sits near the visual center, but not as a modern hero headline. It is an object in a map.
- Ten-plus image nodes float around it in an uneven ring. They vary in size, color, and distance.
- Footer links sit low and small, almost detached from the main composition.

Whitespace:
- Most of the viewport is empty star texture.
- Objects are packed around the center but leave large unstructured margins.
- The whitespace is not calm editorial whitespace; it feels like navigable space.

Typography:
- Labels are tiny, bright, and attached to image objects.
- There is no large paragraph or CTA hierarchy.
- Text behaves as annotation, not marketing copy.

Image / shape / UI handling:
- Every navigational item is a bitmap-like object.
- Image objects are the actual interface, not decoration behind cards.
- The page is closer to an illustrated map than a document.

Information density:
- The first view has many destinations, but very little explanatory prose.
- Density comes from many named objects and their spatial relationships, not from columns of text.

Material feel:
- Star background, saturated object images, low-resolution edges, and small labels create a CD-ROM / web 1.0 materiality.
- The assets feel placed rather than componentized.

Where it differs from generic AI LP:
- No centered eyebrow / H1 / subtitle / CTA stack.
- No equal cards, product screenshots, rounded glass panels, or gradient mesh.
- Navigation is spatial and pictorial, not funnel-shaped.

Where direct mimic would become cheap:
- Copying the star background plus planet icons would become costume retro.
- Centering a giant stylized logo would reproduce the reference too literally.
- Using fake Windows controls or novelty GIFs would turn the experiment into parody.
- Uniform node treatments would lose the important irregularity.

## Avoid list for this build

- No centered hero stack.
- No pill CTA.
- No frosted glass.
- No equal card grid.
- No dark gradient plus floating cards.
- No fake Windows app screen.
- No planet-icon clone of Space Jam.
- No top-only visual strength followed by ordinary sections.

## Aim list for this build

- Make the first viewport read as a strange knowledge kiosk / signal cabinet.
- Use an external licensed diagram as a real central material, not as a tiny decoration.
- Create at least three visual treatments for repeated data nodes.
- Carry the same apparatus language through the full page.
- Keep the page as an information object rather than a sales LP.
